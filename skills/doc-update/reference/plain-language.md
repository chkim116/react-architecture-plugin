# 쉬운 표현 규칙

컨벤션 문서는 **쉬운 한국어**로 쓴다. 읽는 사람이 한 번에 이해 못 하면 규칙으로서 실패다.

## 1. 전문용어를 풀어서 쓴다

영어 아키텍처 용어나 추상적인 한자어를 그대로 쓰지 않는다. 일상어로 바꾼다.

| 쓰지 말 것 | 풀어서 |
|---|---|
| mechanism / input | "공용 설비" / "부르는 쪽이 정할 값" |
| dependency inversion | "인자로 받아서 부르는 쪽이 넘긴다" |
| ambient runtime | "그때그때 정해지는 현재 정보 (예: 오늘 날짜)" |

## 2. 라벨이 모호하면 변형안을 비교로 보여준다

한 단어 라벨이 한 번에 안 와닿으면(예: "값이냐 도구냐") 억지로 하나로 확정하지 않는다.
구체적인 변형안 2~3개를 나란히 보여주고 고르게 한다. 가능하면 실제로 들어갈 문구 그대로 보여준다.

## 3. 예시로 못 박는다

추상적인 규칙 한 줄보다 짧은 예시 하나가 더 명확하다.
"예: external API가 오늘 날짜로 조회한다면 — …"처럼 구체적인 상황을 붙인다.

## 왜

규칙·결정 트리를 어려운 용어로 쓰면 다시 풀어 써야 해서 두 번 일한다.
처음부터 쉽게 쓰고, 애매하면 고르게 하는 편이 빠르고 정확하다.
