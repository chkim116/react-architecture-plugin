# Reference 문서 템플릿

이 레포의 `*/reference/*.md` 문서는 아래 **고정 구조**를 따른다. 새 문서는 이 골격으로 시작하고, 기존 문서를 수정할 때도 이 순서를 유지한다.

목적은 *읽기 최적화*가 아니라 **문서 간 일관성**이다. (A/B 테스트에서 구조를 바꿔도 agent의 추출 정확도는 동일했다 — 그래서 frontmatter·TL;DR 같은 장치는 강제하지 않는다. 같은 모양을 유지하는 것 자체가 목표다.)

## 골격

~~~
# {제목}

{이 문서가 정하는 규칙을 1~2줄로 요약}

## 원칙
- {지시형 핵심 규칙: "~한다 / ~금지"}

## 구현 패턴 / 사용 예시
{대표 코드 예시 — tsx 펜스}

## 네이밍 규칙        (해당될 때만)
| 요소 | 패턴 | 예시 |
| ---- | ---- | ---- |

## 주의사항           (선택)
- {함정·예외}

## 참조               (관련 문서 있을 때만)
- ./other.md 형태의 마크다운 링크
~~~

## 섹션 규칙

- **제목 + 요약**: 필수. 요약은 "이 문서가 무엇을 정하는가"를 한 문장으로.
- **원칙**: 필수. 규칙을 bullet로, 단정형(지시형)으로.
- **구현 패턴 / 사용 예시**: 코드가 있으면 `tsx` 코드 블록으로.
- **네이밍 규칙 / 주의사항 / 참조**: 해당 내용이 있을 때만 둔다. **빈 섹션 금지.**
- 표는 `| 요소 | 패턴 | 예시 |` 형식을 기본으로 한다.
- 링크 표기는 [LINK-CONVENTIONS.md](./LINK-CONVENTIONS.md) 규칙을 따른다.
- **구현 코드 중복 금지(SSOT)**: 공용 모듈의 구현은 `modules/{module}.md`에만 둔다. 주제 문서는 규칙·사용 패턴·링크만.

## 인덱스 + 하위 변형 (`modules/`)

여러 항목을 묶는 문서는 **인덱스 + 하위** 2단으로 나눈다 (`modules.md` ↔ `modules/{module}.md`).

- **인덱스**(`modules.md`): 표로 목록 + "추가 절차" 안내. 구현 코드는 넣지 않는다.
- **하위**(`modules/{module}.md`): 분류·역할 → 구현/사용 규칙 → `## 참조`.
- 인덱스 → 하위는 `./modules/{module}.md`, 하위 → 상위는 `../{file}.md`로 링크한다.
